# RMCL 工程开发路线图

本文档是 `计划文档.md` 的工程实施版。
目标不是继续堆愿景，而是把参考项目能力、当前项目现状、以及接下来真正要做的步骤收束成一条可持续推进的主线。

# 项目约束
- 为了可持续更新，单文件不得超过800行
- 避免独立造轮子，所有功能实现优先从参考项目、同类项目中学习实现

## 1. 文档目标

- 明确参考项目的能力范围，避免开发过程中“想到哪做到哪”。
- 把当前版本从“功能原型”推进到“可持续迭代的启动器内核”。
- 先打通 Vanilla/Fabric 的稳定启动闭环，再向完整启动器和客户端能力扩展。

## 2. 参考项目功能列表

主参考项目是 Prism Launcher。
架构和现代化实现风格可额外参考 Nitrolaunch，但功能基准以 Prism Launcher 为主。

### 2.1 Prism Launcher 的核心能力分组

#### A. 实例系统

- 多实例管理：创建、复制、删除、导入、导出。
- 实例隔离：每个实例独立保存 `mods`、`resourcepacks`、`saves`、日志、实例配置。
- 实例继承与覆盖：全局设置可继承，实例可覆盖 Java、内存、窗口、启动参数等。
- 实例迁移：支持从其他启动器导入数据。

#### B. Minecraft 版本与组件解析

- 拉取 Mojang 版本清单和版本元数据。
- 解析版本继承链 `inheritsFrom`。
- 处理 libraries、natives、assets、logging、main class、JVM/game arguments。
- 按规则系统筛选不同平台、不同 feature 的参数与库。
- 处理 Vanilla、Fabric、Forge、NeoForge、Quilt 等不同加载器。

#### C. 启动流水线

- 启动前校验：实例配置、Java 可用性、版本组件完整性。
- 资源准备：下载缺失文件、校验哈希、解压 natives、准备 logging config。
- 参数生成：生成 JVM 参数、classpath、游戏参数、环境变量。
- 进程生命周期管理：启动、日志采集、退出码处理、错误提示。

#### D. Java 管理

- 自动发现本机 Java。
- 根据 MC 版本选择合适的 Java 主版本。
- 多 Java 安装共存与切换。
- 自动下载建议版本的 Java 运行时。

#### E. 模组与整合包平台

- Modrinth、CurseForge、FTB、Technic、ATLauncher、Packwiz 等平台接入。
- 单 Mod 安装、整合包安装、依赖解析、版本匹配。
- 本地模组启用/禁用、更新、冲突检查。

#### F. 账号系统

- Microsoft 登录。
- 离线账号。
- 多账号切换。
- 头像、皮肤、会话信息管理。

#### G. 日志、诊断与工具链

- 游戏日志实时查看。
- 错误归因与用户提示。
- 崩溃分析辅助。
- 资源下载和启动过程的任务模型。

#### H. 设置系统

- 全局设置和实例级覆盖。
- 路径、代理、下载源、UI、语言、行为设置。
- 可持久化、可迁移、可拓展的配置模型。

#### I. UI 组织方式

- 启动器主页面。
- 实例列表和实例详情。
- 市场页、设置页、账号页、日志页。
- 任务进度与后台状态可视化。

### 2.2 从本地参考仓库可直接借鉴的模块

本地 `example-project` 里，下面这些模块最值得直接借鉴设计思路：

- `launcher/minecraft`
  - 版本格式解析、规则系统、资源组件、实例模型。
- `launcher/launch`
  - 启动任务、启动步骤、日志模型。
- `launcher/java`
  - Java 检测、Java 元数据、安装列表。
- `launcher/modplatform`
  - 市场平台抽象、元数据准备任务。
- `launcher/settings`
  - 设置对象、覆盖机制。
- `launcher/ui/pages`
  - 页面按领域拆分，而不是堆在单个大页面里。

## 3. 当前 RMCL 的现状判断

### 3.1 已经具备的基础

- 前端结构已经成型：首页、实例、市场、账号、设置都有页面骨架。
- Tauri 通信链路已经建立：前端可以调用 Rust 命令，Rust 可以向前端发事件。
- 后端已经拆出若干核心模块：`minecraft`、`resolver`、`download_manager`、`assets_manager`、`java_manager`、`mod_manager`、`process_manager`。
- Fabric 和 Modrinth 已经有原型接入，不是从零开始。

### 3.2 当前最主要的问题

- 启动参数规则评估不完整，导致“能编译但不一定能启动”。
- 实例目录只是配置目录，不是真正的运行时隔离目录。
- Java 选择没有和 MC 版本要求绑定。
- 启动过程缺少明确的中间产物，逻辑散落在命令函数里。
- 缺少统一的可验证出口，容易继续堆功能但不稳。

## 4. 开发总原则

### 4.1 优先级原则

1. 先保证 Vanilla 稳定启动。
2. 再保证 Fabric 走同一条启动主线。
3. 再扩展 Forge/NeoForge/Quilt。
4. 再继续市场、整合包、迁移、插件、客户端增强。

### 4.2 架构原则

- 先抽象“启动域模型”，再堆具体功能。
- UI 不直接拼接业务规则，前端只消费后端整理好的视图数据和任务状态。
- 实例、版本、Java、账号、资源下载分别建模，不要混在一个 command 里硬写。
- 每一步都要有“完成标准”，否则不进入下一步。

### 4.3 工程原则

- 新增能力优先落到模块，不要继续把逻辑堆进 `src-tauri/src/lib.rs`。
- 所有核心流程都要有最少量的单元测试或集成测试样例。
- 关键步骤必须输出结构化日志，便于前端展示和问题排查。
- 能复用公共 pipeline 的地方，不做 Vanilla/Fabric 双套逻辑。

## Step 4：抽出 `ResolvedLaunchSpec`，统一 Vanilla/Fabric 启动流水线

### 8.1 目标

把现在散落在 command 里的启动流程，收敛成一个清晰的中间产物和统一 pipeline。

### 8.2 为什么这是关键

只有抽出 `ResolvedLaunchSpec`，后面 Forge/NeoForge/Quilt、整合包、自定义参数、日志系统、客户端增强才能不失控。

### 8.3 目标结构

建议把启动拆成下面几个阶段：

1. `LoadInstance`
2. `ResolveVersionChain`
3. `ResolveLoader`
4. `ResolveJava`
5. `PrepareAssetsAndLibraries`
6. `PrepareNativesAndLogging`
7. `BuildLaunchSpec`
8. `SpawnProcess`

### 8.4 `ResolvedLaunchSpec` 建议字段

```rust
pub struct ResolvedLaunchSpec {
    pub instance_id: String,
    pub version_id: String,
    pub game_dir: PathBuf,
    pub working_dir: PathBuf,
    pub java_path: PathBuf,
    pub java_major_version: u32,
    pub main_class: String,
    pub classpath: Vec<PathBuf>,
    pub jvm_args: Vec<String>,
    pub game_args: Vec<String>,
    pub envs: Vec<(String, String)>,
    pub natives_dir: PathBuf,
    pub log_config: Option<PathBuf>,
}
```

### 8.5 核心任务

- [ ] 把 `launch_minecraft` 从“大而全 command” 改成 orchestration 层。
- [ ] 新增版本解析层：
  - [ ] 解析 Vanilla 元数据。
  - [ ] 解析 `inheritsFrom`。
  - [ ] 合并 Fabric 元数据。
- [ ] 新增启动规格构建层。
- [ ] 新增进程启动输入对象，而不是直接传 `Vec<String>`。
- [ ] 统一进度事件模型，按阶段汇报。
- [ ] 统一日志事件模型，前端可实时消费。

### 8.6 建议模块拆分

- `src-tauri/src/core/version_resolver.rs`
- `src-tauri/src/core/loader_resolver.rs`
- `src-tauri/src/core/launch_spec.rs`
- `src-tauri/src/core/launch_service.rs`

### 8.7 完成标准

- [ ] Vanilla 和 Fabric 都通过同一条 launch pipeline。
- [ ] `lib.rs` 中只保留 command 注册和 very thin orchestration。
- [ ] 任一阶段失败都能返回明确错误上下文。
- [ ] 前端能看到分阶段进度和日志。
